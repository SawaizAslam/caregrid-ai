# CareGrid AI — Backend

AI-powered healthcare intelligence over ~10k Indian hospital records. Hybrid
search (BM25 + MiniLM/FAISS + metadata filters) + a transparent trust-score
engine with row-level evidence citations, exposed as a small FastAPI
service. Built for the Hack-Nation challenge
[*Serving a Nation*](../data/03_Serving_a_Nation.docx.pdf).

> Built for a hackathon. Optimised for a working demo on CPU, not
> microservices-grade scale.

The backend deploys to **Hugging Face Spaces** via the Docker SDK
(`Dockerfile` in the repo root, `app_port: 7860`). The frontend is
generated by **Lovable** and deployed to **Vercel** — see
[`../frontend/LOVABLE_PROMPT.md`](../frontend/LOVABLE_PROMPT.md) for the
prompt and [`../README.md`](../README.md) for the end-to-end deploy walk-
through.

## Quick start

### 1. Install

```bash
cd HackNation
python -m venv .venv
# Windows: .venv\Scripts\Activate.ps1
# macOS / Linux: source .venv/bin/activate
pip install -r backend/requirements.txt
```

The first `import sentence_transformers` will download the
`all-MiniLM-L6-v2` weights (~90 MB) into `~/.cache/huggingface`.

### 2. Drop the dataset in

Place your hospitals file at one of these locations (first match wins):

```
data/hospitals.csv
data/hospitals.xlsx
data/hospitals.xls
```

The loader is column-tolerant — it accepts variants like `name`/`hospital`
for hospital name, `pin`/`pincode`/`zip` for PIN, etc. (see
`backend/app/data_loader.py::COLUMN_ALIASES`).

A curated `data/pincodes.csv` ships in the repo (state + major-city centroids,
~80 rows). For sharper distances, replace it with a fuller PIN dataset
(e.g. data.gov.in's all-India PIN list) — same schema, same path.

### 3. Build the indexes

```bash
python -m backend.scripts.build_index
```

This produces:

- `data/index/faiss.index`  — FAISS IndexFlatIP over the encoder's vectors
- `data/index/bm25.pkl`     — pickled BM25 keyword index
- `data/index/meta.parquet` — row-aligned metadata + cached trust scores
- `data/index/encoder.pkl`  — only when the TF-IDF/SVD fallback is in use

Subsequent app starts reuse these artifacts.

#### Encoder backends

The embedding stage auto-selects:

1. **`sentence-transformers/all-MiniLM-L6-v2`** (preferred — true 384-d
   semantic embeddings). Downloaded on first use to `~/.cache/huggingface`.
2. **TF-IDF + TruncatedSVD fallback** (sklearn-based, fitted on the corpus,
   384-d L2-normalised). Used when the torch / onnxruntime DLL stack
   can't be loaded — common on bleeding-edge Python builds on Windows
   where the MSVC 2015-2022 Redistributable is missing.

Both backends produce `float32` row-normalised vectors, so FAISS, BM25,
the ranker and the explainer all stay identical — only the encoder swaps.

Force a backend with `CAREGRID_ENCODER`:

```bash
# Force the fallback (useful if MiniLM fails or you want a fast iteration loop)
CAREGRID_ENCODER=tfidf python -m backend.scripts.build_index

# Default = auto-detect (MiniLM, then TF-IDF on failure)
python -m backend.scripts.build_index
```

The active encoder name is reported by `GET /health`.

### 4. Run the API

```bash
uvicorn backend.app.main:app --reload --port 8000
```

Open `http://localhost:8000/docs` for the auto-generated Swagger UI.

### `GET /hospitals/{id}`

Returns the full record for the map detail panel, including coords and
trust breakdown.

### `GET /stats/overview`

Top-line counts: dataset size, facility-type breakdown, average trust score.

### `GET /stats/deserts?top_n=36`

Per-state aggregates ranked by **desert score** (higher = worse). Drives the
Crisis Map choropleth on the frontend.

```json
{
  "states": [
    {
      "state": "Bihar",
      "facility_count": 412,
      "hospital_count": 184,
      "avg_trust_score": 41.2,
      "major_specialty_share": 0.18,
      "hospital_share": 0.45,
      "desert_score": 0.704
    }
  ]
}
```

### `GET /stats/specialty-gaps`

For every major specialty (icu / dialysis / trauma / cardiology / neurology),
counts facilities and ranks states by coverage. Drives the
specialty-gap matrix.

### `GET /stats/contradictions?limit=50`

Returns the rows whose trust breakdown contains a Contradiction rule, with
the citation evidence already attached. Drives the Audit tab.

### `POST /admin/reindex`

Force a fresh build (call this after dropping in a new dataset without
restarting the server).

## How the ranking works (judging cheat-sheet)

For each query the engine produces four signals per hospital, then combines
them. All signals are normalised to `[0, 1]` over the candidate set.

| Signal       | Weight (relevance) | Weight (urgent) | Source                                      |
| ------------ | ------------------ | --------------- | ------------------------------------------- |
| `semantic`   | **0.5**            | 0.1             | MiniLM cosine vs `notes + specialties + equipment` |
| `keyword`    | **0.2**            | —               | BM25 over the same field                    |
| `trust`      | **0.2**            | 0.2             | Pre-computed 0-100 score (see below)        |
| `location`   | **0.1**            | **0.7**         | PIN/radius haversine, or state/district match |

The "urgent" column kicks in when the parser sees `nearest`, `urgent`, `24/7`,
`emergency`, etc. — proximity dominates.

### Trust score (per hospital, 0-100, computed once at index time)

| Rule                                                        | Δ      |
| ----------------------------------------------------------- | ------ |
| Major specialty (ICU / dialysis / trauma / cardio / neuro)  | **+20** |
| Equipment info present (any canonical tag)                  | **+20** |
| Staffing info present (count + role keywords)               | **+20** |
| Contradiction: claims major specialty, no equipment AND no staff | **−30** |
| > 2 of {specialties, equipment, staff, notes, location} missing | **−10** |
| Rich-tag bonus (cap +20, used as a tie-breaker)             | +0..+20 |

Final score is clamped to `[0, 100]` and accompanied by a `trust_breakdown`
list so the UI can show *why*.

### Query understanding (rule-based, no LLM)

The parser is deterministic and offline:

- **State**: longest-substring match against the 28+8 state/UT list, with
  abbreviation aliases (`up` → Uttar Pradesh).
- **PIN**: `\b\d{6}\b`.
- **Radius**: `within (\d+) km`, or default 30 km when "nearest" is used.
- **Specialties / requirements**: synonym-aware vocab lookup
  (`o2`→oxygen, `vent`→ventilator, `kidney`→dialysis, etc.).
- **Sort**: `distance` if any urgency keyword fires, else `relevance`.

The parsed JSON ships back inside every `/search` response under
`query_understood`, so judges (and the frontend) can see exactly what was
understood from the user's text.

## Tuning knobs (`backend/app/config.py`)

- `EMBEDDING_MODEL` — swap out for a different sentence-transformer.
- `TOP_K_FAISS` / `TOP_K_RESULTS` — candidate / result counts.
- `W_SEMANTIC`, `W_KEYWORD`, `W_TRUST`, `W_LOCATION` — relevance weights.
- `W_URGENT_LOC`, `W_URGENT_TRUST`, `W_URGENT_SEM` — urgency weights.
- `CAREGRID_DATA_DIR` env var — override the data directory.
- `CAREGRID_CORS_ORIGINS` env var — comma-separated CORS origins.
